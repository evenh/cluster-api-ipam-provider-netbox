# LEARNINGS.md

## Durable Learnings

### Repository workflow
- `LEARNINGS.md` must be consulted before each meaningful step.
- Only durable, reusable findings should be persisted here.
- GitHub Actions in this repo should stay thin and delegate behavior to `Makefile` targets; CI formatting checks should use `make fmt-check` rather than mutating the tree with `make fmt`.

### Project decisions
- The provider-specific pool APIs use `ipam.cluster.x-k8s.io/v1alpha1`.
- Cluster API integration should target the latest Cluster API IPAM contract in use by the project.
- The repository target is Go `1.26.4` (`go.mod`); CI and local test flows should derive the toolchain from `go.mod` via `actions/setup-go`'s `go-version-file`, not a hardcoded version.
- The Dockerfile's builder base image must track or exceed `go.mod`'s `go` directive, or `GOTOOLCHAIN=auto` silently downloads the required toolchain at build time (extra network dependency, slower builds). The builder stage must also be pinned `FROM --platform=${BUILDPLATFORM} golang:X` so multi-arch builds cross-compile via `GOARCH` instead of QEMU-emulating the builder itself (without the pin, buildx pulls the *target*-arch base image for the builder stage too).
- `gcr.io/distroless/static:nonroot` already bundles `ca-certificates` and full `tzdata` (verified by extracting the image filesystem); do not add either explicitly or switch base images to get them.
- NetBox integration is implemented with repo-owned JSON clients; do not reintroduce `go-netbox` unless there is a concrete maintenance reason to accept generated-client coupling again.
- Direct YAML imports in this repo should use `go.yaml.in/yaml/v4`; do not add new direct uses of the deprecated `gopkg.in/yaml.v3` module.
- Shared controller plumbing in this repo should go through `pkg/reconcileutil.ControllerBase`; avoid duplicating `Client`/`Scheme`/`Recorder` fields or recorder helper methods per reconciler.
- Pool CRDs expose an optional `spec.clusterName`; the pool reconciler mirrors it into the standard `cluster.x-k8s.io/cluster-name` label so pools can participate in `clusterctl move`.
- Container images publish to `ghcr.io/evenh/cluster-api-ipam-provider-netbox` (repo is private; the GHCR package defaults to private too â€” pulling it elsewhere needs `imagePullSecrets` or making the package public). CI shape follows telekom/cluster-api-ipam-provider-infoblox's pattern: a reusable `build-image.yml` (`workflow_call`, multi-arch amd64+arm64) called by `docker-build-check.yml` (PR-only, no push), `container-image.yml` (push to `main` -> `latest`+`sha-`), `container-image-manual.yml` (`workflow_dispatch`, build+push from any branch/commit on demand), and `release.yml` (`v*` tags -> semver tags + a draft GitHub Release with `dist/ipam-components.yaml`+`metadata.yaml` attached, via `make build-installer`).
- For local/manual debugging against a real cluster without paying for NetBox+kind bring-up on every run: `make e2e-up` provisions once (disables the testcontainers ryuk reaper, execs the compiled manager binary directly rather than `go run` so its PID is trackable, persists connection details to `.cache/e2e/state.json`), `make e2e-test-reuse` (`E2E_REUSE=1`) runs the chainsaw scenarios against that standing environment repeatedly (~30s vs ~5min), `make e2e-down` tears it down explicitly.

### Confirmed environment facts
- The repository started effectively empty and must be bootstrapped from scratch.
- Local tooling already available includes `go`, `kubebuilder`, `controller-gen`, and `kustomize`.
- Commands that compile or inspect Go packages must use a repo-local `GOCACHE`, because the default macOS Go build cache path is not writable in the sandbox.
- `kubebuilder` successfully writes scaffold files before failing on dependency resolution, so it can still be used as a file generator if dependency updates are handled separately.
- `controller-gen paths=./...` will also traverse temporary nested scaffold modules under the repo root and can fail on their unrelated module state; generation commands should target the real package trees or those temp dirs should be removed intentionally.
- The repository pins `controller-gen` to `v0.20.1`; generated CRDs and RBAC should be emitted with `bin/controller-gen`, not an older globally installed binary.
- Upgrading this repo from controller-runtime `v0.22.x` to `v0.23.x` does not require source changes for the `v0.23.0` release-note breakages, because the project does not currently use controller-runtime event recorders or webhook validator/defaulting interfaces.
- On controller-runtime `v0.23.x`, manager-scoped event recording should use `mgr.GetEventRecorder(...)` and `k8s.io/client-go/tools/events.EventRecorder`; `GetEventRecorderFor` is deprecated.
- `sigs.k8s.io/cluster-api-ipam-provider-in-cluster v1.0.3` is not compatible with CAPI `v1.12.3`; it still imports removed `cluster-api` `v1beta1` packages.
- `sigs.k8s.io/cluster-api-ipam-provider-in-cluster/pkg/ipamutil` (latest `v1.1.0`) is a genuinely reusable library â€” its package doc documents this, and telekom/cluster-api-ipam-provider-infoblox and nutanix-cloud-native/cluster-api-ipam-provider-nutanix both import it directly instead of forking. Its `ProviderAdapter`/`ClaimHandler` interfaces are identical in shape to this repo's `pkg/ipamutil`. Not adopted here because `v1.1.0` pins CAPI `v1.11.2`/controller-runtime `v0.21.0`, behind this repo's `v1.13.3`/`v0.24.1` â€” adopting today would mean downgrading. Revisit if upstream catches up; our fork also has `Ready`-condition and event-recording logic upstream `v1.1.0` lacks.
- The Cluster API IPAM `v1beta2` contract (`sigs.k8s.io/cluster-api/api/ipam/v1beta2`) defines a `Ready` condition on `IPAddressClaim` with reasons `AllocationFailed`/`PoolNotReady`/`PoolExhausted`, but core CAPI does not set it â€” only the IPAM provider's own reconciler can, since CAPI has no controller for claims itself. `pkg/ipamutil.ClaimReconciler` sets it via `sigs.k8s.io/cluster-api/util/conditions.Set`; the object's `SetConditions`/`GetConditions` methods are enough for `patch.Helper` to auto-diff and patch it, no extra wiring needed.
- All NetBox HTTP calls in this repo should set the shared custom `User-Agent` string from `internal/netbox.UserAgent`.
- NetBox `v4.5.x` bootstrap creates only v2 API tokens; the e2e harness must set `API_TOKEN_PEPPER_1`, derive the token key from container logs, and use `Authorization: Bearer nbt_<key>.<secret>`. Runtime clients should accept both legacy v1 tokens and v2 `nbt_...` tokens.
- In this sandbox, envtest cannot bind a local control-plane port (`listen tcp 127.0.0.1:0: bind: operation not permitted`), so controller and envtest suites require elevated execution to run here.
- In `cmd/main.go`, controller manager flags must be parsed into the same config instance that is returned; registering flags against a local copy and parsing later silently ignores runtime overrides like `--health-probe-bind-address=0`.
- Cluster API provider repositories still use `metadata.yaml` with `apiVersion: clusterctl.cluster.x-k8s.io/v1alpha3`; the release series should advertise the provider contract separately (`v1beta2` here).
- NetBox `v4.3-3.3.0` serves `/api/` as unauthenticated `403 Forbidden`; the e2e readiness check should treat that as healthy rather than waiting for `200`.
- NetBox bootstrap API tokens created via `SUPERUSER_API_TOKEN` must fit the backing `varchar(40)` limit.
- NetBox rejects unknown IP address custom field names in allocation payloads; any custom fields referenced by pool metadata or claim annotations must already exist in NetBox, so the e2e harness must seed them explicitly.
- Chainsaw configuration requires `metadata.name`; `skip delete` behavior is controlled by the CLI `--skip-delete` flag rather than `spec.cleanup`.
- Chainsaw (`v0.2.15`) is a `go.mod` `tool` dependency here (`go get -tool github.com/kyverno/chainsaw@...`), invoked as `go tool chainsaw`, not a Makefile-downloaded binary in `bin/` â€” mirrors kartverket/skiperator's convention. This pulls a large transitive dependency tree into `go.sum` (helm, kuttl, cloud SDKs); that bloat is expected and does not affect the shipped manager binary, only build tooling.
- Moving Go code out of `_test.go` files (e.g. to make it importable by a separate `package main` tool) loses golangci-lint's built-in `_test.go` path exclusion (`bodyclose`, `cyclop`, `dupl`, `errcheck`, `funlen`, `goconst`, `gocognit`, `gosec`, `noctx`, `testpackage`, `wrapcheck` in this repo's `.golangci.yml`). Extracted test-infra `.go` files need their own matching exclusion rule, or the same code that was clean as a `_test.go` file suddenly produces dozens of findings.
- Testcontainers' `ryuk` reaper kills every container/network from a test session the moment the controlling process disconnects â€” including on clean exit, not just crashes. A standalone provisioning tool that must leave containers running after it exits (e.g. `test/e2e/cmd/e2eup`) must set `TESTCONTAINERS_RYUK_DISABLED=true` before creating any docker resources.
- A CEL `XValidation` rule that references an unbounded string field inside an unbounded array (e.g. `self.cidr.contains('/')` with no `MaxLength` on `cidr` and no `MaxItems` on the containing list) can exceed the Kubernetes API server's CEL cost budget ("estimated rule cost exceeds budget by factor of more than 100x") even though the rule itself looks trivial. Fields referenced by CEL rules need explicit `MaxLength`/`MaxItems` bounds.
- A namespaced CRD reconciled by a single cluster-scoped manager (cluster RBAC on Secrets) can leak cross-namespace secret reads if it exposes an optional "secret ref" field that allows specifying an arbitrary namespace â€” worth a CEL check (or plain code check) forbidding that field outside cluster-scoped types. Confirmed and fixed for `NetBoxIPPool.spec.connectionSecretRef.namespace` (only `GlobalNetBoxIPPool`, which has no owning namespace, may set it).
- This provider's actual NetBox read/write split: prefixes and the claim-UID custom field (`cluster_api_claim_uid` by default) must already exist in NetBox â€” the provider only reads/resolves them and errors if the custom field is missing. Tags are created on demand. IP address records are the real write path: one is created per accepted `IPAddressClaim` and deleted when the claim is deleted. The provider never imports/discovers pre-existing NetBox IP addresses into Kubernetes.
